# Next.js App Router, SSR scoped to doc detail + sitemap

## Status

Accepted

## Decision

BUSCASAM's frontend is a Next.js application (App Router, latest stable) deployed as a single Node process on the same on-prem VM as FastAPI + TEI + Postgres + worker (ADR-0001, ADR-0002, ADR-0003). A reverse proxy fronts both processes on one origin: HTML routes go to Next, `/api/*` goes to FastAPI. SSR is restricted to the two pages that need crawler-visible HTML — `/docs/[id]` (público documents are SEO targets per SPEC §Visibilidad) and `/sitemap.xml` / `/robots.txt`. Every other page (home, `/buscar`, browse-by-hierarchy, `/publicar`, `/perfil`, `/notificaciones`) is client-rendered. The frontend never owns business logic: Server Components on the SSR path forward the request `Cookie` header to FastAPI on `http://localhost:8000` and let the visibility predicate from ADR-0001 §9 decide what to return. A typed client generated from FastAPI's OpenAPI schema (ADR-0003 §9) is the only seam between the two processes; no BFF, no hand-written API mirror.

## Context

SPEC requires public documents to be discoverable by search engines while restricting `interno` and `privado` documents to authenticated viewers — a mixed-SSR-and-CSR surface, not a uniform one. The same SPEC asks for reproducible search URLs (already a CSR-friendly shape: URL → state → fetch), a browse-by-hierarchy navigation tree (for humans, not crawlers), an authenticated publishing flow with multi-step file upload, and an in-app notifications channel. ADR-0001 puts the architecture on one VM at 16 GB with bounded headroom after Postgres (~4 GB), TEI (~2 GB), FastAPI workers (~1 GB), and the async indexer (~0.5 GB). ADR-0003 commits to FastAPI's auto-generated OpenAPI as the API contract and pins a "one seam" discipline between processes (the queue is the only seam between API and worker; OpenAPI is the only seam between frontend and API). The team is small and may rotate, on UNSAM-managed infra, Python on the backend and now TypeScript on the frontend.

## Considered options

- **SvelteKit.** Smaller bundles, cleaner mental model (`+page.server.ts` / `+page.svelte`), native form actions that match the publish flow well. Rejected on hiring pool / continuity: the rotating-academic-team risk is higher with Svelte than with React in Argentina, and SvelteKit's ecosystem (OpenAPI clients, accessible component libraries, docs) is thinner than React's by a meaningful margin in 2026.
- **Remix / React Router v7 framework mode.** Form-and-loader-first, web-platform-aligned. Rejected after the React Router merger: the positioning is now muddled, the community-investment line has flattened, and Next.js covers the same ground with more docs and more contributors.
- **SPA React + Vite.** No Node in prod, simplest to operate. Rejected: doc detail must be crawler-renderable for SEO, and either you add a per-publish prerender pipeline (couples publish worker to a Node CLI) or you serve a JS-only shell that Google may index poorly. The "no Node" saving evaporates as soon as one page needs SSR.
- **Next.js Pages Router.** Mature, smaller surface, simpler caching model. Rejected on directional grounds: Pages Router has been legacy-track since Next 13.4 (May 2023); by the time a new contributor joins in 2026+, App Router is "Next.js" and Pages Router is the historical detour. Choosing it now is a forced migration in 2-3 years.
- **BFF in Next.js route handlers (`app/api/*` proxying to FastAPI).** Rejected: breaks ADR-0003 §9's "OpenAPI is the contract" invariant by inserting a TypeScript mirror that drifts. Same-origin reverse proxy gets the encapsulation benefit (browser never reaches a foreign origin) without the duplicated surface.
- **Live SSR for every route, including search and browse.** Rejected: search has no SEO requirement (Q1 of the grill), browse pages are navigation for humans and a `sitemap.xml` does the crawler's job better. SSR everywhere would put authenticated CSR-grade traffic onto Node for no payoff.
- **Pre-render público doc detail at publish time, no live SSR.** Rejected: every edit, co-author confirmation, soft-delete, and metadata update would need a re-render step, coupling the publish worker to a Node render binary that can drift from the running frontend. Two consistency stories (DB row + cache file) where one suffices.
- **CSS Modules + hand-rolled UI primitives, or Tailwind alone without shadcn.** Rejected: SPEC needs accessible combobox (autocomplete), dialog (publish modal), dropdown (filters), and threaded comments — primitives that get accessibility wrong easily. shadcn/ui (Radix-based, copy-pasted source) bakes this in without a runtime npm dep that can break on update.

## Architecture decisions locked by this ADR

1. **Framework.** Next.js, latest stable, **App Router** (`app/` directory). Pinned to a specific minor in `frontend/package.json`. Standalone build output (`output: 'standalone'`) for the Docker image (ADR-9 territory).
2. **Process topology.** One Node process for Next.js, deployed on the same VM as FastAPI + TEI + worker + Postgres. Memory budget: ~200 MB. A reverse proxy (Caddy or Nginx, locked in ADR-9) routes `/api/*` to FastAPI on `:8000` and everything else to Next on `:3000`. TLS terminates at the proxy.
3. **SSR surface.** Exactly two routes render server-side: `app/docs/[id]/page.tsx` and the file-routed `app/sitemap.ts` + `app/robots.ts`. Every other route is a Client Component with `'use client'` at the top. The split is enforced by code review, not tooling; the principle is "SSR exactly the SEO surface."
4. **Cookie-forwarding SSR.** The `/docs/[id]` Server Component reads the incoming `Cookie` header via `next/headers` and forwards it on the internal `fetch('http://localhost:8000/docs/{id}', { headers: { cookie }, cache: 'no-store' })`. FastAPI applies the visibility predicate from ADR-0001 §9 — anonymous gets `público` only, UNSAM-authenticated gets `público + interno`, author gets all three. The Server Component never makes a visibility decision; on 404 from FastAPI it calls `notFound()`. Sitemap query is hard-coded to `WHERE visibility = 'publico' AND NOT soft_deleted`.
5. **No BFF.** No `app/api/*` route handlers that proxy or wrap FastAPI endpoints. The only exception is Next's framework-internal handlers (e.g., the SSO callback if it must land on the same origin — to be settled in ADR-5). Browser fetches go through the reverse proxy directly to FastAPI; Server Components fetch FastAPI on `localhost`.
6. **API client.** `openapi-typescript` regenerates `frontend/src/api/schema.d.ts` from FastAPI's `/openapi.json` as a CI step. `openapi-fetch` (~6 KB runtime) is the typed client; same function call shape in Server Components (`cache: 'no-store'`) and Client Components. TanStack Query is added per-feature for client-side caching/invalidation (search results, comments, favorites) — not blanket.
7. **No SSR caching at MVP.** Doc-detail Server Component uses `cache: 'no-store'`. `Cache-Control: private, no-store` on the response. Postgres single-row PK lookup is sub-millisecond; Node + FastAPI round-trip is the steady-state cost. If anonymous crawler traffic later proves to be the bottleneck, add proxy-level caching keyed on absence-of-cookie with a `revalidate` hook called from the publish worker — explicit, not implicit.
8. **Forms.** React Hook Form + Zod for client-side validation. Forms submit directly to FastAPI through the reverse proxy. No Next.js Server Actions: they recreate the BFF pattern this ADR rejects (§5). File upload (publish flow) is `multipart/form-data` straight to FastAPI, which returns 202 and enqueues per ADR-0003 §4. Zod schemas are hand-written for the ~5-6 mutating forms (not codegen'd from Pydantic).
9. **Styling and primitives.** Tailwind CSS for styling; shadcn/ui for accessible primitives (Radix-based, copy-pasted source under `frontend/src/components/ui/`). No runtime CSS-in-JS (incompatible with RSC). No second component library.
10. **Repo layout.** Monorepo at the repo root: `backend/` (existing Python/Poetry tree from ADR-0003) and `frontend/` (new Node/pnpm tree). Top-level CI runs both pipelines; the OpenAPI codegen job boots FastAPI, hits `/openapi.json`, regenerates `frontend/src/api/schema.d.ts`, and fails the PR if the diff is uncommitted. Deploy units remain independent (two processes, one image, separate entrypoints).
11. **Toolchain.** pnpm as package manager (`packageManager` field in `package.json`); Node LTS pinned via `.nvmrc` (Node 22 at time of writing). Next's default ESLint config + Prettier. Vitest for unit/component tests; Playwright for E2E smoke (search happy path, publish happy path, doc-detail render). Type-check (`tsc --noEmit`) is a required CI gate.
12. **i18n.** No i18n framework. UI strings are Spanish, written inline. SPEC's only language requirement is Spanish for end-user content; agent-facing files stay in English per `CLAUDE.md`.
13. **Client state.** Local component state (`useState`, `useReducer`) plus TanStack Query for server state. No Zustand, Redux, Jotai, or other global client store at MVP. If a cross-component client state need emerges (e.g., notifications dropdown opened from multiple places), introduce the smallest scoped Context, not a global store.

## Consequences

- **The "OpenAPI is the only seam" invariant is now load-bearing on both sides.** ADR-0003 §9 stated it; this ADR consumes it. A backend PR that changes a response shape without regenerating `schema.d.ts` fails CI in the frontend; a frontend PR that uses a field FastAPI no longer returns fails type-check. The contract has no other enforcement.
- **The visibility predicate is centralised in SQL, by design (ADR-0001 §9), and the frontend cannot bypass it.** Server Components forward cookies and render whatever FastAPI returns. A bug in the SSR path that adds a "show all docs" toggle still goes through the WHERE clause — same gravity as the search chokepoint in ADR-0003 §3.
- **Two languages in the repo (Python + TypeScript) and two test stacks.** Onboarding cost is now per-language. Mitigation: monorepo with shared CI, single PR for cross-cutting changes, OpenAPI codegen as the bridge.
- **Node memory adds to the 16 GB VM budget.** ~200 MB Node + ~30-50 MB Next.js working set per concurrent SSR request. Comfortably within budget at MVP traffic. The binding constraint remains TEI + Postgres HNSW; Node is noise.
- **App Router's `'use client'` boundaries are unforgiving.** New contributors will hit "you can't use useState in a Server Component" or "you imported a Server-only module from a Client Component" within the first session. Onboarding doc is one short page; the SSR surface is small enough that most files have `'use client'` at the top and the question doesn't arise.
- **CSR-by-default means no progressive enhancement.** Search, publish, comments, favorites all require JavaScript. Acceptable for an authenticated academic platform; flagged so we don't surprise ourselves later.
- **Doc-detail SSR cost is unbounded by user count.** Every public doc page-view (including crawler) renders fresh. Bounded in practice by total document count × crawl frequency. Revisit if `lexical_fallback_rate`-style metrics on Node show real pressure.
- **No SSR caching at MVP is a deliberate "until we measure" choice.** When/if added, the invalidation contract must run through the queue (ADR-0003 §4): publish/edit/soft-delete worker emits a job that calls a Caddy purge endpoint or a Next `revalidatePath`. Don't add caching without that invalidation path; stale public pages are an academic-credibility failure.
- **The reverse-proxy decision is anticipated here but locked in ADR-9.** This ADR depends on the proxy existing; ADR-9 owns the configuration. If ADR-9 picks a topology without a proxy (unlikely), §2 and §4 of this ADR must be revisited.
- **The SSO/auth shape is open (ADR-5).** This ADR commits to "the request `Cookie` header carries the session"; whether that cookie is signed-JWT, opaque-server-session, or something else is ADR-5's call. The cookie-forwarding mechanic in §4 works for all common shapes.
- **shadcn/ui is "owned source," not a dependency.** Component updates are manual (re-copy from upstream when needed). Trade-off: no auto-breaking on dep update, but also no auto-fixing on security patch. Watch shadcn's changelog for primitives we adopt.
